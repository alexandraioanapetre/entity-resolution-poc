04 — Data Quality Issues
⚠
CSV Parsing Corruption — Input Row Keys & Country Fields
~52 input rows have their input_main_country field populated with description text, NAICS codes, or industry labels (e.g. "Custom Computer Programming Services", "518210", "All Other Telecommunications"). These appear to be caused by a CSV delimiter conflict — the semicolons in long_description or generated_description shifted columns. Also affects input_row_key which contains sentence fragments. These rows are effectively unresolvable without raw data correction.
52
⚡
Country Mismatch — Input vs. Matched Entity
58 matched rows have a different country code for the input vs. the matched candidate. Most are intentional (regional subsidiary matched to global parent, e.g. Salesforce.com Singapore → Salesforce US). However, a minority represent ambiguous matches where a global HQ was used as a fallback because the local subsidiary was missing from the engine (e.g. Shields Environmental Group Ltd UK → Shields Environmental US). These should be flagged for client review.
58
🔁
Duplicate Veridion IDs — Multiple Inputs Mapped to One Entity
19 Veridion entity IDs were matched to 2 or more different input companies. Several are legitimate (e.g. a company + its branch both matching the same parent: Telenor ASA and Telenor Real Estate AS → both matched to "Telenor"). Others represent inputs that are actually distinct subsidiaries from different countries mapped to the same global parent due to engine gaps (e.g. 3 STEP IT AS Norway + 3 Step IT Sweden AB → 3Step IT Group Finland). Clients consuming this data may receive duplicate records for the same Veridion entity.
19
💸
Revenue Outliers — Scale Normalization Issue
The top revenue values are extremely large and appear to be in inconsistent units. Salling Group shows 10.1 trillion (likely 10.1 billion DKK reported in øre or wrong scale). Aviat Networks shows 244 billion (likely $244 million USD). These are consistent with a unit normalization bug — some values may be in local currency smallest units (øre, cents) while others are in whole currency. A conversion/normalization pass is required before any revenue-based analysis or client delivery.

07 — Curation Recommendations
01
Fix CSV parsing — re-export with proper quoting
The ~52 rows with corrupted country/key fields were caused by unescaped semicolons inside text fields (descriptions, tags). Re-export the source data with all text fields wrapped in double quotes, or switch to a non-conflicting delimiter (pipe | or tab). Before re-export, validate that all multi-value fields use a consistent internal separator that doesn't conflict with the CSV delimiter.
02
Normalize revenue to a single currency & unit (USD, whole)
Revenue values are inconsistently scaled. Values like 10.1 trillion (Salling Group) and 244 billion (Aviat Networks) indicate a mix of local currency units, cent/øre denominations, and USD. Apply a normalization pipeline: (1) identify currency per entity from main_country_code, (2) convert smallest units to whole units, (3) apply FX conversion to USD. Flag modelled vs. extracted revenue with different confidence tiers.
03
Add a match_confidence field to distinguish match quality tiers
Not all "Yes" matches are equal. Recommend introducing a confidence tier: HIGH = exact address + name match; MEDIUM = global parent matched with confirmed regional presence (phone/email/location); LOW = global parent used as fallback with no regional confirmation. This lets clients downstream filter by confidence and prioritize HIGH-tier records for outreach.
04
Flag country mismatches as "parent match" rather than direct match
58 matches involve the input company being from a different country than the matched Veridion entity. Add a boolean field is_parent_match (or match_type = "subsidiary_to_parent") so the client knows the exact entity they submitted is not in the engine — only the parent is. This is critical for clients who need jurisdiction-specific data (e.g., local regulatory compliance, correct legal entity for contracts).
05
Deduplicate Veridion ID reuse across inputs before client delivery
19 Veridion IDs are used for 2+ input rows. For the client output, either (a) deliver one record per Veridion ID and list all input companies that map to it, or (b) deliver one row per input but add a duplicate_entity_flag so the client can deduplicate themselves. Without this, the client may count the same company twice in their analytics or outreach lists.
06
Enrich or exclude 80 no-contact stub matches
80 matched entities have no website, email, or phone. These are essentially empty shells with only a name and geography. For client delivery, consider: (1) triggering a supplementary enrichment pass on these entities, or (2) downgrading them to "low confidence — uncontactable" status. Delivering these as full matches would inflate match rate without delivering actionable business intelligence.
07
Investigate engine coverage gap for regional subsidiaries
~110 of the 215 unmatched rows are cases where the global parent exists in the engine but the local subsidiary does not. This is the single largest source of unmatched records. Recommend a targeted engine coverage audit for the most common gap patterns: Nordic-country branches of global tech companies (SAP Danmark, TCS Sverige, Telenor LDI, Tech Mahindra Malaysia, etc.). A subsidiary indexing pass on these ~15 companies alone would recover ~30–40 unmatched rows.
